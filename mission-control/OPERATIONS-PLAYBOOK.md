# Benchmark Mission Control — Operations Playbook

This is the operating manual for running all client sites from one place. Open the board with the
**Mission Control** desktop shortcut (or `mission-control/index.html`). Update it by talking to Claude
in the **Claude - Mission Control** terminal (launched in `benchmark-sites/`).

## How the board works
- One data file per site: `mission-control/data/{fgs,brm,te,jsc}.js`. **Each site's terminal owns its
  own file** — never have two terminals edit the same file (last save wins).
- `data/performance.js` is auto-generated by `pull-performance.py` (DataForSEO). Don't hand-edit it.
- Item schema: `type` = Blog | Location Page | Service Page | Page · `status` = idea | drafting |
  ready | scheduled | published | live | open · `date` ISO · optional `url` (link to open/approve).
- To change anything, just tell Claude: *"schedule an FGS post on percolation tests for Jul 25,"*
  *"mark the Citrus location page ready,"* *"add a BRM low-hanging-fruit page for ITIN loans."*

## Google Business Profile workflow (you run all the GBPs)
- **Every blog publish → one GBP post** that links the article (drives the post + sends a relevance
  signal to the profile). Claude can draft the GBP blurb when a post goes live.
- **Plus 2 standalone GBP posts/week** per profile: an offer, a project photo, a tip, a season note.
  Photos from the gallery / `OneDrive\...\Graphics` are good fuel.
- GBP is the #1 local-pack lever — more so than the website for "near me" searches.

## Social media — COMBINED here, not a separate tool (recommended)
Keep social in Mission Control as a per-site **Promotion** track rather than a separate calendar:
most social/GBP posts are repurposed from the blog/project that spawned them, so they belong next to
it; one cadence, one place, no second app to maintain. Only BRM + TE run social, so a standalone
system isn't worth it. (Promotion lanes are a near-term roadmap item — see ROADMAP.md.)

## Content refresh policy (don't just publish — maintain)
- **Low-hanging fruit first.** Pages ranking **#4–15** (shown in each site's Performance panel) move to
  page 1 / top 3 with small on-page work — better title/intro, an FAQ, internal links, freshened copy.
  This beats writing net-new for traffic-per-hour. Current standouts: BRM "usda lenders florida" #12
  (1,900/mo), JSC "barndominium builder in florida" #6 (1,000/mo), TE "tire express services" #9.
- **Cornerstone/money pages:** review every **6 months**; refresh if rank slipped or it's >12mo stale.
- **Cost/stat/year posts** (e.g. FGS cost guide "2026"): refresh every **January**.
- **Evergreen explainers:** review every 12–18 months.
- **Trigger-based always beats calendar:** refresh the moment a page drops in position or a competitor
  passes it (caught by the monthly performance pull).

## Routines
**Weekly (in the Mission Control terminal):**
1. Open the board. Check each site's Publishing Schedule — anything due this week drafted?
2. Fill any **open slots** with a topic (low-hanging-fruit keywords are ready-made topics).
3. Confirm GBP posts done (2/site + any blog-linked).
4. Move cards as work progresses (idea → drafting → ready → scheduled → published).

**Monthly:**
1. Re-run the performance pull: `python mission-control/pull-performance.py` (refreshes top pages +
   low-hanging fruit from DataForSEO; ~$0.05).
2. Re-pull domain traffic if wanted; update the `traffic` numbers per site file.
3. Review refresh policy — flag anything cornerstone that's gone stale.
4. Commit + push so the laptop + history stay current.

## Commands (run from `benchmark-sites/`)

```
powershell -File sync-to-repo.ps1              # mirror workspaces into the repo
git add -A && git commit -m "mc: ..." && git pull --rebase origin main && git push
```
